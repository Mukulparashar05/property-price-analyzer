# Dual-Mode House Price Chatbot Design

**Goal:** Add a dual-mode Streamlit experience where users can either enter all house features manually or switch to a chatbot that accepts free-form prompts, silently completes missing model inputs with Ollama, and returns a house-price prediction generated by the existing trained ML pipeline.

## Product Summary

The application will support two user flows:

1. **Manual Mode**
   Users fill out the existing structured fields directly and receive a prediction from the trained scikit-learn pipeline.

2. **Chat Mode**
   Users type free-form requests such as "I want a house near the ocean" or "suggest me a location for a family home." Ollama interprets the request, completes any missing model fields internally, and produces a full prediction-ready payload. The ML model then generates the final price estimate. The user sees the chatbot response and predicted price, but does not see the hidden filled-in values.

## User Experience

### Manual Mode

- Keep the current form-based workflow for users who want precise control.
- Preserve the existing feature fields required by the model:
  - `longitude`
  - `latitude`
  - `housing_median_age`
  - `total_rooms`
  - `total_bedrooms`
  - `population`
  - `households`
  - `median_income`
  - `ocean_proximity`
- Keep input validation and direct prediction behavior.

### Chat Mode

- Add a toggle or segmented control at the top of the app to switch between `Manual Mode` and `Chat Mode`.
- In chat mode, replace the form with a conversational interface using Streamlit chat components.
- Let the user provide:
  - partial feature information
  - non-technical requests
  - location suggestions
  - broad house-buying prompts
- The chatbot should behave like a real assistant:
  - understand intent from natural language
  - infer missing fields internally
  - return a conversational answer
  - always include a final predicted house price after hidden autofill completes the missing model inputs

## Core Behavior

### Prediction Source of Truth

The scikit-learn model stored in `model.pkl` remains the only prediction engine. Ollama must never replace the trained ML model for numeric price prediction.

### Ollama's Role

Ollama will be used for two responsibilities:

1. Parse free-form chat input into model-relevant meaning.
2. Fill missing house features so the ML model always receives a complete payload.

### Hidden Autofill Rule

When users omit fields in chat mode, the chatbot must silently complete the missing values. Those assumptions remain hidden from the user. The user sees only the chatbot reply and the final result, not the full completed input record.

## Architecture

Split the application into focused units instead of keeping all logic in one Streamlit file.

### `app.py`

- Render the top-level Streamlit UI.
- Provide the mode toggle.
- Render manual form controls.
- Render chat history and chat input.
- Call prediction and chat helper functions.
- Display final prediction and assistant response.

### `predictor.py`

- Load `model.pkl`.
- Expose a prediction function that accepts the full nine-field payload.
- Return a float price prediction.

### `ollama_chat.py`

- Send chat messages and schema instructions to Ollama.
- Ask Ollama to return:
  - a natural-language assistant reply
  - a full set of completed model inputs
- Handle model host configuration and connection failures.

### `validation.py`

- Validate numeric ranges and categorical values before prediction.
- Provide backend fallback defaults for values that Ollama leaves invalid or incomplete.

## Data Flow

### Manual Mode Flow

1. User enters all fields.
2. Backend validates the payload.
3. Valid payload is passed directly to the ML model.
4. App displays predicted price.

### Chat Mode Flow

1. User sends a free-form message.
2. The app sends Ollama:
   - the user message
   - recent chat history
   - instructions describing the required nine ML fields
3. Ollama returns:
   - a natural-language reply
   - a completed structured payload for the model
4. Backend validation checks the completed payload.
5. Invalid or missing values are repaired using safe defaults.
6. The final full payload is sent to the ML model.
7. The app shows the chatbot response and predicted price.

## Missing-Value Strategy

Because the user explicitly wants hidden autofill, the backend must guarantee a complete model input even when the chat request is vague.

### Numeric defaults

Use median fallback values derived from the training dataset for:

- `longitude`
- `latitude`
- `housing_median_age`
- `total_rooms`
- `total_bedrooms`
- `population`
- `households`
- `median_income`

The implementation should compute or store those medians once in backend code so chat mode uses deterministic defaults instead of ad-hoc guesses.

### Categorical default

Use `INLAND` as the categorical fallback for `ocean_proximity` if Ollama does not provide a valid category.

### Validation repair

If Ollama returns values outside allowed ranges, the backend must clamp or replace them before model prediction instead of crashing.

## Chatbot Response Policy

The chatbot should respond conversationally, but remain grounded in the house-price use case.

### Expected response types

- House-buying guidance
- Location suggestions
- High-level recommendations
- A predicted price generated from the completed payload on every chat turn that reaches inference

### Constraints

- Do not expose hidden autofilled fields.
- Do not claim exact model internals or exact feature importance unless computed separately.
- Do not pretend the ML model is the LLM.
- Keep the chatbot answer user-friendly and practical.

## Error Handling

### Ollama unavailable

- Chat mode should show a friendly failure message if Ollama is unreachable.
- Manual mode must still work even if Ollama is down.

### Invalid Ollama output

- If Ollama returns malformed structured data, retry once with a stricter schema prompt.
- If the second attempt still fails, use backend defaults plus a fallback assistant response.

### Invalid model input

- Prevent the prediction call from receiving invalid categorical values or impossible numeric values.
- Show a safe user-facing error only if the backend cannot recover.

## State Management

- Use Streamlit session state to preserve:
  - selected mode
  - chat history
  - latest prediction
  - latest assistant response
- Keep manual mode state independent from chat mode state so switching modes does not corrupt the current session.

## Testing Strategy

### Manual mode tests

- Validation still rejects impossible inputs.
- Prediction still works with a complete manual payload.

### Chat mode tests

- Ollama output is parsed into the expected schema.
- Missing values are autofilled before model prediction.
- Invalid Ollama values are repaired by backend validation.
- Chat mode degrades gracefully when Ollama is unavailable.

### Smoke tests

- Keep `ollama_test.py` as the simple connectivity check for local Ollama access.

## Non-Goals

- Re-training the ML model
- Replacing the ML model with an LLM
- Showing the hidden autofilled feature payload to the user
- Building a multi-page application

## Output

The final result should be a Streamlit app with a visible toggle between manual entry and chatbot interaction, backed by the current trained model and a local Ollama assistant that silently completes incomplete user requests before prediction.
